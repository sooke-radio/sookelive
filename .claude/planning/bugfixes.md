- **Live preview doesn't refresh in the admin panel iframe** (reported 2026-07-15, testing via `10.0.0.142:3006` in cc-container). Root cause fixed 2026-07-16, live browser re-check still needed:
  - Confirmed fixed as part of this: Shows was missing from `generatePreviewPath.ts`'s `collectionPrefixMap`, so its preview URL resolved to `undefined/<slug>` (commit 8b085eff).
  - Confirmed still correctly wired: `LivePreviewListener` (`RefreshRouteOnSave` from `@payloadcms/live-preview-react`) is mounted in all three frontend templates (`posts/[slug]`, `shows/[slug]`, `[slug]`); `/next/preview/route.ts` already awaits `draftMode()` (Next 16-compatible).
  - **Fixed**: `generatePreviewPath.ts` was preferring `NEXT_PUBLIC_SERVER_URL` (hardcoded to `http://localhost:3000` in dev `.env`) over the incoming request's actual host, so the preview iframe/link was built pointing at `localhost:3000` even when the admin was accessed via the LAN IP `10.0.0.142:3006` — wrong origin for that browser, fails silently client-side (no server log entry, matches what was observed). Flipped the precedence so `req.host` (verified via `payload`'s `createPayloadRequest`, derived straight from the incoming request URL) wins whenever present, falling back to `NEXT_PUBLIC_SERVER_URL` only if it's missing. Confirmed `stg`/`prd` are unaffected: Caddy's `reverse_proxy` doesn't rewrite the `Host` header, so `req.host` already equals the public domain there too — same result as the old `NEXT_PUBLIC_SERVER_URL`-first behavior, just also correct for LAN-IP dev access. Covered by `src/utilities/generatePreviewPath.spec.ts`. CORS, sitemap generation, and `syncAzuracast`'s self-call still read `NEXT_PUBLIC_SERVER_URL` independently and are untouched.
  - Still needed: a live browser check via `10.0.0.142:3006` (or equivalent LAN address) confirming the preview iframe actually loads and refreshes now — couldn't be done from this sandbox (no LAN browser access, admin is auth-gated).
- **`bin/sync-prd-to-stg.sh`: prd's media didn't show up on stg's frontend after a sync** (found 2026-07-16 while testing the script for real). Diagnosed and fixed:
  - Not the cause: the media file copy itself (`docker exec` + `tar`, container-to-container) works correctly — verified prd and stg had identical file lists (310 files, same names) after a real run.
  - Root cause: `copy-db.sh`'s `mongorestore --drop` swaps stg's whole database directly, which completely bypasses Payload's application layer — so none of the collections' `afterChange` revalidate hooks fire, and stg's Next.js cache keeps serving pages rendered *before* the sync (stale/missing images), indefinitely, regardless of what's actually in the DB or on disk now. Restarting the container doesn't fix this either — Next's route cache persists on disk across restarts, only container recreation or an explicit revalidation clears it.
  - Fix: added `POST /api/revalidate-all` (`src/endpoints/revalidateAll.ts`, same `isAuthenticatedOrCronSecret` auth as the jobs/playlists-sync endpoints) which calls `revalidatePath('/', 'layout')` to bust every page in one call. `bin/sync-prd-to-stg.sh` now calls it against stg as its final step (via `docker exec` + Node's built-in `fetch`, using the container's own `CRON_SECRET` — no curl/wget dependency). Also added a "Revalidate All Pages" button on the Media admin list view (`src/components/RevalidateAll`) for manual use whenever staleness shows up outside this specific sync flow — this has happened before and wasn't previously fixable without a redeploy.
  - Verified: endpoint tested end-to-end against the running dev server (401 with no/wrong auth, 200 with the real `CRON_SECRET`); covered by `tests/int/revalidateAllEndpoint.int.spec.ts`. Not verified: an actual live prd→stg sync run with the new revalidate step (needs the real server).
